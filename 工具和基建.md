# Oculai 前沿基建技术选型调研报告

> 记录时间：2026-05-26  
> 选型原则：前沿优先、Agent-Native、中文环境、中国候选人 + 全球证据  
> 风险偏好：激进创新，允许后续降级/替换

## 0. 核心选型立场

Oculai 是一个 **Agent-Native 中文人才寻聘系统**。所有基建选型必须服务于以下前提：

1. **Claude Code 主 Agent 做所有决策**。任何基建工具不能成为“第二个 Planner”或“第二个大脑”。
2. **中文环境优先**。数据源、搜索引擎、浏览器工具必须覆盖中文生态（百度、知乎、CSDN、Gitee、BOSS 直聘、脉脉等），同时从全球源获取证据。
3. **中国候选人 + 全球证据**。候选人来自中国/中文生态，但 GitHub、arXiv、Semantic Scholar、OpenAlex 等全球源用于验证和丰富候选人档案。
4. **前沿优先，不保守**。不因某个工具当前不够成熟就排除，优先考虑工具的理念、架构方向和 Agent-Native 潜力。允许后续因稳定性问题降级。
5. **激进但受控**。反爬/隐身浏览器可以写入架构，但只是公开资料 fallback；所有登录态、外部副作用必须有审批闸门。

## 1. 数据源与搜索

### 1.1 学术数据源

| 数据源 | 接入方式 | Star/活跃度 | 适合 Oculai 的理由 | 风险 |
|---|---|---|---|---|
| **arXiv** | `blazickjp/arxiv-mcp-server` (2.8k★, Apache-2.0, Python) 或直接 OAI-PMH API | 高 | MCP 原生、搜索/下载/分析一体；学术论文是最可靠的人才证据来源 | API 有速率限制，大范围搜索需要缓存 |
| **Semantic Scholar** | 直接 REST API + 自研 Oculai source connector | 官方 API 稳定 | 作者 h-index、引用量、ORCID、机构归属，直接用于候选人质量评估 | 需 API key；部分中文论文覆盖不全 |
| **OpenAlex** | `cyanheads/openalex-mcp-server` (5★, Apache-2.0) 或直接 API | 官 API 稳定 | 270M+ 出版物，覆盖论文→作者→机构→研究主题图谱 | MCP 包 star 低但官方 API 成熟 |
| **DBLP** | 直接 REST API + 自研 Oculai source connector | 官方稳定 | CS 领域最完整的文献目录，最适合技术人才寻聘 | 无中文全文；API 相对简单 |

**建议接入路线**：  
- P0：直接封装 arXiv / Semantic Scholar / OpenAlex / DBLP 官方 API 为 Oculai SourceConnector。  
- P1：按需加载 `blazickjp/arxiv-mcp-server` 作为 arXiv 快捷探索接口。

#### 1.1.1 学术会议论文（高价值新增）

**2026 年是学术会议论文公开获取的分水岭。**

| 渠道 | 开放状态（2026） | 内容规模 | Oculai 价值 |
|---|---|---|---|
| **ACM Digital Library** | **全面开放**（2026.1.1） | 60 万+ 篇论文，含所有会议 proceedings，年增 1.8 万篇 | CS 全领域会议论文，最高质量的技术人才证据 |
| **arXiv** | 一直开放 | 200 万+ 预印本，CS 各子领域 | AI/ML/CV/NLP 等方向的最新研究，预印本即时发现活跃研究者 |
| **OpenAlex** | 开放 API | 2.7 亿+ 出版物，含 proceedings-article 类型 | 跨学科会议论文元数据，可按 venue/年份/主题过滤 |
| **DBLP** | 开放 API + XML dump | CS 领域最全的会议/期刊目录 | 作者-论文-会议关联图谱，最精确的 CS 人才索引 |
| **LIPIcs (Dagstuhl)** | CC BY 4.0 开放出版 | 逐年增长，被 CCF 推荐会议广泛采用 | 完全开放的论文全文 |
| **OpenReview** | 开放 API / 可爬取 | ICLR/NeurIPS 等顶会投稿和审稿记录 | 发现正在活跃投稿的研究者；审稿意见也是能力证据 |

**程序化获取方式：**

```text
1. 会议论文发现
   → OpenAlex API 按 type:proceedings-article + publication_year + venue 搜索
   → DBLP API 按 conference + year 获取论文列表
   → Semantic Scholar 按 venue 过滤

2. 全文获取
   → ACM DL（现在免费）: 直接通过 DOI 获取 PDF
   → arXiv: OAI-PMH / API 获取 PDF
   → LIPIcs: 直接下载
   → Unpaywall API: 通过 DOI 查找 OA 版本
   → OpenReview: 爬取或 API 获取

3. 论文解析
   → MinerU / Docling: PDF → Markdown / JSON
   → 提取：作者列表、机构、关键词、引用、致谢（合作者/导师关系）
```

**对人才寻聘的关键价值：**
- 会议论文作者列表直接暴露**活跃研究者及其合作网络**
- 论文主题和关键词揭示**技术方向和深度**
- 引用数据（Semantic Scholar）反映**学术影响力**
- 第一作者/通讯作者区分**贡献层级**
- 会议级别（CCF A/B/C）间接反映**研究水平**
- 审稿记录（OpenReview）展示**同行评价和学术声誉**

**建议接入路线：**  
- P0：OpenAlex + DBLP + Semantic Scholar 三路交叉发现会议论文和作者。  
- P0：ACM DL / arXiv / LIPIcs 作为全文获取源。  
- P1：OpenReview 爬取（顶会投稿者发现）。

### 1.4 中文全域搜索

**关键发现：百度官方在 ClawHub 发布了 7 个 OpenClaw Skills。**

`ide-rea` 是百度在 OpenClaw 生态的官方组织账号，已发布：搜索、学术搜索、百科、网盘、地图等 7 个 Skills。其中 `baidu-search` 全球下载量排名第一（88k+），远超其他搜索引擎 skill。

| 数据源 | 接入方式 | 下载量 | 适合 Oculai 的理由 | 集成方式 |
|---|---|---|---|---|
| **百度网页搜索** | `ide-rea/baidu-search` (MIT-0, Python, v1.1.4) | 88k+ | 百度官方 AI 搜索 API (BDSE)；省 token 的官方接口，无需爬虫；支持 freshness 过滤、count 控制 | OpenClaw skill → Oculai MCP wrapper |
| **百度学术搜索** | `ide-rea/baidu-scholar-search-skill` (v1.1.0, Shell) | 5.6k+ | 中英文学术论文、期刊、会议论文搜索；支持分页+摘要；中文科研人才发现核心入口 | OpenClaw skill → Oculai MCP wrapper |
| **百度百科** | `ide-rea/baidu-baike-data` | - | 结构化中文知识；候选人背景信息验证 | OpenClaw skill → Oculai MCP wrapper |
| **知乎** | 暂无成熟 MCP/Skill | - | 中国技术/学术讨论主阵地，个人主页含教育/职业/研究方向 | OpenCLI + Crawl4AI |
| **CSDN / 掘金 / 博客园** | 暂无 MCP | - | 中文技术博客，技术栈和项目经验证据 | OpenCLI + Crawl4AI |
| **微信公众号 / B站** | 暂无 MCP | - | 中文内容生态重要来源 | OpenCLI 或其他采集方案 |

**为什么 baidu-search + baidu-scholar-search 比 Evilran/baidu-mcp-server 更好：**

| 对比维度 | Evilran/baidu-mcp-server | ide-rea/baidu-search |
|---|---|---|
| 开发者 | 个人 | **百度官方** |
| 生态验证 | 25 GitHub star | **88k+ 下载**，全球第一 |
| 技术路线 | BaiduSpider 网页抓取 | 百度 AI 搜索 API (BDSE, 官方接口) |
| 法律风险 | 中（网页抓取，反爬对抗） | **低**（官方 API） |
| API Key | 不需要 | 需要百度千帆 BAIDU_API_KEY |
| 学术搜索 | 无 | **有**（baidu-scholar-search，5.6k 下载） |
| 百科 | 无 | **有**（baidu-baike-data） |
| 安全审计 | 无 | **已通过** |
| 时效过滤 | 无 | 支持 freshness（pd/pw/pm/py 或日期范围） |

**接入方式分析：**

这两个是 OpenClaw Skills，不是 MCP Servers。但接入 Claude Code / Oculai 完全可行：

```text
方式 A（CLI 直接调用）
  Claude Code Bash tool → python3 skills/baidu-search/scripts/search.py '{"query":"...","count":10}'
  优点：最快，零封装
  缺点：裸 JSON 不够语义化

方式 B（Oculai MCP 封装，推荐）
  Oculai MCP Server → oculai.search_baidu(query, count, freshness)
                    → oculai.search_baidu_scholar(keyword, page, include_abstract)
  内部调用 OpenClaw skill 脚本
  优点：语义化工具、审计日志、rate limit、结果归一化

方式 C（Oculai Python Backend 封装）
  Oculai Python backend → BaiduSearchProvider.search()
  内部通过 subprocess 或直接调用百度千帆 API
  优点：和 Exa/Tavily Provider 接口统一
```

**推荐：方式 B + C 混合。** Oculai MCP 暴露 `oculai.search_baidu` 和 `oculai.search_baidu_scholar` 两个领域工具，底层封装百度官方 Skill 脚本，保持与 GitHub/Exa 等 provider 一致的接口风格。

**对比 Evilran/baidu-mcp-server：**

之前推荐 `Evilran/baidu-mcp-server` 是因为不了解百度官方已有更成熟方案。现在 `ide-rea/baidu-search` + `baidu-scholar-search` 在各方面都更优，且是百度官方出品、88k 下载验证过的方案。`Evilran/baidu-mcp-server` 不再需要，除非作为免 API Key 的紧急 fallback。

**建议接入路线**：  
- P0：`ide-rea/baidu-search`（官方，88k 下载，MIT-0）→ Oculai `search_baidu` 工具。  
- P0：`ide-rea/baidu-scholar-search-skill`（官方，5.6k 下载）→ Oculai `search_baidu_scholar` 工具。  
- P1：`ide-rea/baidu-baike-data` → Oculai `lookup_baidu_baike` 工具。  
- P0：OpenCLI + Crawl4AI 覆盖知乎、CSDN、掘金、博客园、公众号。  
- P2：`Evilran/baidu-mcp-server` 仅作为免 API Key 紧急 fallback（如果百度千帆 API 不可用）。

### 1.5 个人主页、博客与社交账号（高价值新增）

个人主页和博客是最被低估的人才证据来源。一个研究者的个人主页通常包含：论文列表、项目经历、合作者、教学经历、获奖、开源项目链接、最新研究方向和未发表工作。技术博客则展示其工程思维深度和表达能力。

#### 1.5.1 发现渠道

| 渠道 | 提取方式 | 典型 URL 来源 | 内容价值 |
|---|---|---|---|
| **DBLP** | API 返回 author homepage 字段 | 学术个人主页 (.edu / .ac.cn) | 论文、合作者、研究组、教学 |
| **Semantic Scholar** | Author API 含 homepage 字段 | 学术/机构主页 | 引用指标、研究领域 |
| **GitHub** | REST API `user.blog` 字段 | 技术博客、个人站点 | 开源项目、技术栈 |
| **arXiv** | 作者页有时含个人链接 | 学术主页 | 预印本、研究方向 |
| **OpenAlex** | Author API 含 affiliations + 可交叉查 | 机构主页 | 单位、合作网络 |
| **百度搜索** | `baidu-mcp-server` 搜索 "{姓名} 个人主页" | 各种中文主页 | 中文信息全覆盖 |
| **知乎 / CSDN / 掘金** | OpenCLI + Crawl4AI 抓取 | 中文技术社区 | 技术文章、社区影响 |

#### 1.5.2 抓取与解析

```text
发现阶段 → DBLP / Semantic Scholar / GitHub API 获取 homepage URL
         → baidu-mcp-server 搜索补充中文个人主页/博客/社交账号

抓取阶段 → 学术主页 (.edu/.ac.cn)     → Crawl4AI → Markdown → 结构化提取
         → 技术博客 (GitHub.io/自有域名) → Crawl4AI → Markdown
         → 中文博客 (CSDN/掘金/博客园)  → OpenCLI + Crawl4AI
         → 知乎 / B站                    → OpenCLI + 登录态浏览器
         → 微信公众号                     → OpenCLI 或其他采集方案

解析阶段 → MinerU / Docling: HTML → Markdown → 提取
         → 结构化抽取：研究方向、项目、论文、合作者、任职经历、获奖
```

#### 1.5.3 对人才寻聘的关键价值

- 个人主页论文列表可能比数据库**更新、更完整**（含 in-preparation / under-review）
- 技术博客反映**技术深度、表达能力和持续学习**
- 社交账号活动反映**社区影响力、技术关注点和交流能力**
- 合作者网络和项目经历反映**团队协作和领导力**
- 获奖和开源贡献反映**认可度和社区贡献**

#### 1.5.4 建议接入路线

- P0：DBLP + Semantic Scholar + GitHub API 三路获取 homepage URL，封装为 Oculai `get_person_web_presence` 工具。
- P0：`baidu-mcp-server` 搜索补充中文主页/博客/社交账号。
- P0：Crawl4AI 统一抓取和解析非登录态页面。
- P1：OpenCLI 处理需要登录态的中文平台（知乎完整内容、微信公众号等）。

### 2.3 选型建议

```text
主力抓取  → Crawl4AI (本地, Apache-2.0) + Firecrawl (cloud, AGPL)
            二者互补：Crawl4AI 用于大批量低成本抓取，Firecrawl 用于高难度 JS 页面
```

## 3. 浏览器自动化

### 3.1 选型矩阵

| 工具 | 仓库 | Star | License | 语言 | 形态 | 反检测 | Agent Skill | MCP | 中文场景 |
|---|---|---|---|---|---|---|---|---|---|
| **Playwright CLI** | `microsoft/playwright-cli` | 10.7k | Apache-2.0 | TypeScript | CLI + SKILLS | ✗ | ✓ | ✗ | ✓ |
| **Playwright MCP** | `microsoft/playwright-mcp` | 33k | Apache-2.0 | TypeScript | MCP Server | ✗ | ✗ | ✓ | ✓ |
| **Agent Browser** | `vercel-labs/agent-browser` | 34k | Apache-2.0 | Rust | Native CLI | ✓ 轻量 | ✓ | ✗ | ✓ |
| **OpenCLI** | `jackwener/OpenCLI` | 22.7k | Apache-2.0 | JavaScript | CLI + Chrome Extension | ✗ | ✓ | ✗ | **最佳** |
| **Stagehand** | `browserbase/stagehand` | 22.8k | MIT | TypeScript | SDK | 依赖 Browserbase | ✓ | ✗ | ✓ |
| **browser-use** | `browser-use/browser-use` | 95k | MIT | Python | Python SDK | ✓ | ✓ | ✓ | ✓ |
| **Patchright** | `Kaliiiiiiiiii-Vinyzu/patchright` | 3.3k | Apache-2.0 | TypeScript | Playwright fork | **最强** | ✗ | ✗ | ✓ |
| **Camoufox** | `daijro/camoufox` | 8.7k | MPL-2.0 | C++ | Firefox fork | **最强** | ✗ | ✗ (有第三方 MCP) | ✓ |

### 3.2 关键分析

#### Playwright CLI vs Playwright MCP

两个都是微软官方。**Playwright CLI** 的 README 明确宣称 coding agents 用 CLI + skills 比 MCP 更 token-efficient，因为 MCP 会把大量 tool schema 和 accessibility tree 灌入 context。**Playwright MCP** 的价值在需要持续浏览器会话、accessibility introspection、长自主工作流的场景。

Oculai 策略：**二者都接入，CLI 优先，MCP 补充**。

- CLI 用于确定性的 navigate/click/type/extract/screenshot。  
- MCP 用于需要 accessibility tree、迭代推理页面的复杂场景。

#### Agent Browser（Vercel Labs）

34k star 的 Rust 原生 CLI，专为 AI agent 设计。token-efficient CLI 形态 + Chrome for Testing 集成 + skill 支持。是 Playwright CLI 的有力竞品。Oculai 应将其作为备选/并行方案。

#### OpenCLI（中文场景核心）

OpenCLI 有三个独特价值：
1. **把任何网站变成 CLI**，内置适配器覆盖知乎、B站、小红书、Reddit、Twitter 等。
2. **复用已登录 Chrome**，通过 Chrome Extension Bridge 在用户真实浏览器 session 中运行 —— 这是处理 LinkedIn、知乎、BOSS 直聘等需要登录态的平台的正确方式。
3. **CLI Hub** —— 可以把 `gh`、`docker`、`opencli browser` 等注册为统一 CLI。

Oculai 策略：**OpenCLI 作为中文场景浏览器自动化的核心 CLI**。

#### Stagehand

AI + 代码混合的浏览器自动化框架。`act()` 用自然语言操作，`extract()` 用自然语言抽取结构化数据，`agent()` 执行多步骤任务。自带 auto-caching 和 self-healing。

但它的 agent 调用 LLM 做动作决策，这会在 Claude Code 之外引入 LLM 调用。

Oculai 策略：**P1 接入，仅用于特定的浏览器自动化场景，不作为主决策层**。所有 `act()` / `agent()` 调用必须记录完整 log 和 evidence。

#### browser-use

95k star，生态最强但本质是 **browser agent**。它自己调用 LLM（Claude/Gemini/GPT）决定浏览器动作。这直接冲突 Oculai "Claude Code 做所有决策" 的原则。

Oculai 策略：**暂不作为核心链路**。可以作为实验性 subagent backend，用 `browser-use` 的 Agent 执行具体浏览器操作，但策略和判断仍然在 Claude Code 主 Agent。通过 `Saik0s/mcp-browser-use` (932★) 的 MCP 封装调用。

#### Patchright & Camoufox（反爬前沿）

两个最强的反检测浏览器方案：
- **Patchright**：Playwright 的 undetected fork，针对性修补 Runtime.enable / Console.enable / Command Flags 等已知检测点。3.3k star 但专门性极强。
- **Camoufox**：Firefox fork，专为 AI Agent 设计，headless、undetectable、optimized to run at scale。8.7k star。支持 Playwright 兼容的 Python 接口。

Oculai 策略：**激进但受控**。默认抓取走 API / Crawl4AI / Playwright CLI / OpenCLI。遇到 bot wall 时按级升级：Crawl4AI 反爬模式 → Scrapling → Patchright → Camoufox。所有升级后的抓取结果必须标记 `stealth_mode: true` 和 `backend: patchright/camoufox`。

### 3.3 浏览器策略汇总

```text
默认链路  → Playwright CLI (确定性的 Agent 调用)
          + Playwright MCP (持续会话/accessibility 推理)
          + Agent Browser (Rust CLI 备选)

中文登录态→ OpenCLI (核心)
          + Chrome profile 复用
          + 所有登录态操作必须审批闸门

AI 辅助  → Stagehand (P1, 仅限特定场景)
          + browser-use (实验性, 仅作 subagent backend)

反爬升级  → Crawl4AI 反爬 → Scrapling → Patchright → Camoufox
          (逐级升级，标记 stealth_mode)
```

## 4. 文档解析

### 4.1 选型矩阵

| 工具 | 仓库 | Star | License | 语言 | 输入格式 | 输出格式 | MCP | VLM | OCR |
|---|---|---|---|---|---|---|---|---|---|

### 4.2 关键分析

**MinerU** 是国内最强的文档解析方案（来自 OpenDataLab），特别适合 Oculai 中文场景：
- 109 语言 OCR 识别
- VLM + OCR 双引擎
- PDF/DOCX/PPTX/XLSX/图片/网页全覆盖

- 原生 LangChain / LlamaIndex / Crew AI / Haystack 集成

- 许可证友好（MIT）

## 6. AI Agent 工具平台

以下工具本质上是 **Agent 运行时/浏览器**，不是 Oculai 底层能力面。它们可以接入，但不能替代 Claude Code 主 Agent 的决策。

### 6.1 工具分析

| 工具 | 本质 | Star | 与 Oculai 的关系 | 建议 |
|---|---|---|---|---|
| **browser-use** | Browser Agent SDK | 95k | 竞争中存在 LLM 调用 | 实验性 subagent backend |
| **BrowserOS** | Agentic Browser 产品 | 11k | 完整 Agent 运行时 | ⚠️ 暂不接入核心 |
| **OpenBrowser** | Browser Agent Toolkit | 9.5k | Agent 自主浏览器 | ⚠️ 暂不接入核心 |
| **AIPex** | Browser Agent | 1.2k | 类似 browser-use | 暂不考虑 |

### 6.2 关键判断

**这些工具的强大之处 —— Agent 在浏览器内做规划和决策 —— 恰恰是 Oculai 不该依赖它们的原因。** Oculai 的架构原则是 Claude Code 主 Agent 做决策，Python/MCP 层只做执行。如果浏览器工具内部又调用 LLM 决定 "下一步点哪里"、"如何填表"、"是否提交"，就会形成隐式的多重决策链。

唯一可用的方式：如果 Oculai 主 Agent 显式委托一个 browser-use subagent "在 Google Scholar 搜索 X，收集前 20 条结果，返回结构化列表"，把 browser-use 当作一个确定性 worker（而非自主 Agent），那可以接受。但此时 browser-use 的 LLM 调用只是为了实现"怎么操作浏览器"这个底层动作，不是做"搜谁、搜什么、怎么评估"这些领域决策。

## 9. 关键风险与应对

| 风险 | 等级 | 应对 |
|---|---|---|
| Exa/Tavily 中文覆盖不足 | 中 | `baidu-search` + `baidu-scholar-search`（百度官方 API）作为中文主力；Exa/Tavily 只做英文/全球补充 |
| Crawl4AI 与 Firecrawl 功能重叠 | 低 | 互补：Crawl4AI 本地低成本大批量，Firecrawl 高难度 JS 页面 |
| OpenCLI Chrome Extension 安全风险 | 高 | Extension 权限最小化；登录态操作全部审批闸门；敏感操作记录完整 audit log |
| MinerU 自定义许可证 | 中 | 确认商业使用条款；Docling MIT 作为本地 fallback |
| Patchright/Camoufox 法律合规 | 高 | 默认不使用；只处理公开页面；避开拓聘平台和私隐数据；所有抓取标记 source+method |
| LinkedIn/BOSS 反爬风险 | 高 | 不自动化高频抓取；只读；审批闸门；遵守 robots.txt 和 ToS |
| 百度搜索反爬 | 中 | 百度 provider 内置速率控制；按需升级到反爬链路 |
| browser-use 引入第二个决策层 | 中 | 只用做实验性 subagent backend；主 Agent 保留所有策略/判断/评估决策 |
